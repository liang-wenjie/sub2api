# 多角度图片逐张回传设计

## 背景

启用“多角度分别生成”后，GPT 图片模型和 Gemini 图片模型都会把多个角度放入一个插件 job。GPT 在该 job 内顺序调用上游接口，直到全部角度结束才写回历史结果；Gemini 将多个角度作为一个 batch 的多个 item，但只在 batch 总体完成后读取 item。两条路径都会使已完成图片被延迟展示。

## 目标

- GPT 多角度分别生成时，保持单个插件 job，但每个角度完成时立即显示。
- Gemini 多角度分别生成时，batch 未完成但已有 item 完成时立即显示对应图片。
- 多个角度仍在一条用户消息和一条助手消息中展示。
- 取消、失败、重试和历史恢复保持与现有分组任务语义一致。
- 不修改主站服务、数据库结构或对外 API 路径。

## 设计

### 统一增量结果协议

所有图片模型都保持一个插件 history job。job 处于 `pending` 时允许在 `result.images` 中持续追加已完成图片，且 status 接口在 `pending` 响应中原样返回这些图片。前端只需处理一种状态：`pending + result.images`，先显示新图片，再继续轮询。

每张已归档图片以稳定来源 ID 记录在 job 结果中，后续轮询依据该 ID 去重。该协议可供未来支持异步 batch、串行调用或流式回调的模型复用，不需要为模型类型引入新的前端状态或 API 路径。

### GPT 本地任务

GPT 多角度仍由一个后台本地任务和一个 history job 承载。后台按角度调用上游接口；每个角度成功后立即归档图片、补上 `variant_label`，将其追加到该 job 的 `result.images` 并持久化。job 在还有未完成角度时保持 `pending`；全部成功后标记 `succeeded`。若后续角度失败，保留已完成图片并记录失败角度，job 以已有结果的部分成功语义结束。

取消继续使用现有本地任务上下文：取消后不再追加新图片，已归档图片保留在记录中。

### Gemini batch

Gemini 多角度仍发送一个带 `variants` 的 batch 请求。后端在每次 batch 状态查询时，即使 batch 仍在 `pending`，也读取 batch items。对于当前 job 的已完成 item：读取图片内容、归档后追加到 `result.images`；使用 item custom ID 去重，防止后续轮询重复下载或重复写图。

batch 仍未完成时，job 保持 `pending`。全部 batch 完成时沿用现有终态收尾逻辑，保留已归档图片并补齐其余图片或失败角度。

### 前端轮询

收到 `pending` 状态时，前端先将响应中的图片按角度标签或槽位位置写入空槽位，再继续轮询。已填充槽位不被后续 pending 响应覆盖。

## 错误与边界

- GPT 单个角度失败不阻塞其他角度展示。
- Gemini item 查询不返回已完成图片时，维持现有等待行为。
- Gemini 增量归档失败时，status 请求返回错误并由前端按现有重试轮询处理；不将整个 job 误标为成功。
- 已归档 Gemini item 在后续轮询和最终完成时不会重复下载或产生重复图片。

## 测试

- 后端：GPT 两个角度在同一 history job 中执行；第一张完成后 job 保持 pending 且 `result.images` 含该图，第二张完成后 job 成功；取消后不追加后续图片。
- 前端：pending 状态响应携带一张图片时，槽位立即显示并继续轮询。
- 后端：pending Gemini batch 的 items 包含一个完成 item 时，status 保持 pending 但结果含该图片；下一次相同 items 不重复读取内容；batch 完成时保留该图片并补齐其余图片或失败标签。
